# 101Hakata Mingle - 受付アプリ 要件定義

## 1. プロジェクト概要

イベントコミュニティ「101Hakata Mingle」の受付業務を効率化するWebアプリケーション（PWA）。

**解決したい課題:**

- 受付の人員削減（4人 → 3人想定）と時間短縮
- 手作業・紙管理の廃止
- 参加者データの蓄積・活用

**アプローチ:**

- QRチェックインによる受付効率化
- メンバーシップによる特別感の創出
- メール通知によるイベント告知・リマインド

---

## 2. ユーザー種別と権限

| 種別 | 認証 | できること |
| --- | --- | --- |
| ゲスト | 未ログイン（メール + 規約同意） ※名前は自動生成 | イベント閲覧・参加登録・参加キャンセル |
| 参加者（会員） | ログイン済み | 全機能（マイページ・参加履歴・セルフチェックイン等） |
| 運営 | ログイン済み + 管理者ロール | 管理画面（イベント作成・参加者管理・チェックイン処理） |

- 認証方式: メール/パスワード
- 運営権限: Supabaseのユーザーテーブルのロールカラムで区別（Row Level Securityで制御）
- ゲストは以下の機能が制限される:
  - **マイページ・参加履歴の閲覧**（ログイン必須のため）
  - **セルフチェックイン**（QRコードスキャン後の本人確認ができないため、運営による手動チェックインで対応）

---

## 3. 利用フロー

### 参加者フロー（会員）

```
ログイン/登録 → 規約同意(初回) → イベント一覧 → イベント選択 → 決済 → 参加確定
                                                                          ↓
                                        [イベント当日] 会場のQRをスキャン → セルフチェックイン完了
```

1. ログイン（未登録の場合は新規登録）
2. 規約同意（初回のみ / 写真撮影に関する同意を含む）
3. イベント一覧から参加したいイベントを選択
4. 決済（Stripe Checkout / 当日現金払い ※イベントごとに事前決済限定も可）
5. 参加確定 → メール通知受信
6. イベント当日: 会場に掲示されたQRコードを参加者が自分のスマホでスキャン → セルフチェックイン

### ゲストフロー

**Path A: 事前登録**

```
イベント一覧 → イベント選択 → 「ゲストとして参加登録」 → メアド・規約同意 → 決済 → 参加確定
                                                                        ↓
                                    [イベント当日] 会場で運営にメアドを伝える → 運営が手動チェックイン
```

1. イベント一覧から参加したいイベントを選択
2. 「ゲストとして参加登録」を選択
3. メールアドレス・規約同意（写真撮影同意含む）を入力 ※名前は自動生成（Guest123等）
4. 決済（当日現金払いのみ。stripe_onlyイベントはゲスト非対応）
5. 参加確定 → メール通知受信（キャンセル用リンク含む）
6. イベント当日: 会場で運営にメアドを伝える → 運営がメアド検索で手動チェックイン

**Path B: 当日登録（QRから）**

```
会場でQRスキャン（未ログイン） → ゲスト登録画面 → メアド・規約同意 → 決済 → 参加確定
                                                                 ↓
                                             その場で運営が最終チェックイン
```

1. QRをスキャン → ログイン画面表示
2. 「ゲストとして登録」リンク → ゲスト登録画面へ
3. メールアドレス・規約同意を入力 ※名前は自動生成
4. 決済（当日現金払いのみ）
5. 運営がゲスト情報を確認 → 現金受取 → 最終チェックイン

**補足:**

- **事前決済限定イベント（stripe_only）**: 会員のみ対象。Stripe Checkoutでの決済が完了していないとセルフチェックインできない。ゲスト非対応
- **現金払い可イベント（stripe_and_cash）**: 会員はStripe事前決済 or 当日現金、ゲストは当日現金のみ
- **ゲストは当日現金払い**: チェックイン時に運営が現金を受取 → paymentStatusを'paid'に更新 → 最終チェックイン
- **会員のセルフチェックイン**: Stripe事前決済した会員のみ。QRスキャン後、運営が最終確認を行う（ゲート無し対応）
- **会員の当日現金払い**: セルフチェックイン不可。運営が手動チェックイン画面から確認 → 現金受取 → 最終チェックイン
- **ゲスト→会員引き継ぎ**: ゲストで参加登録したメールアドレスで後に会員登録した場合、自動的に過去の参加履歴を紐付け

### 運営フロー

1. イベント作成・公開時の設定:
   - 決済方式（stripe_only / stripe_and_cash）
   - 定員・返金期限・キャンセル期限
   - 担当スタッフを3名程度割り当て（event_staff）
2. 参加申込の通知をメールで受信（運営共有アドレス）
3. 管理画面で参加者一覧・支払いステータス・参加者詳細を確認
4. イベント当日:
   - 会場にチェックイン用QRを掲示
   - Stripe事前決済の会員: QRスキャン → スタッフが目視確認 → 最終チェックイン
   - ゲスト・当日現金ユーザー: スタッフ受付画面でメアド検索 → 現金受取（必要に応じて）→ 最終チェックイン
   - 定員到達時は「受付終了」と表示。キャンセル枠が出ると自動で再受付
   - 必要に応じて `allowOverCapacity` で定員超過を許可
5. イベント終了後:
   - 最終的なチェックイン数を確認
   - 売上を確認（Stripe + 現金）
   - 監査ログで運営操作を確認可能

## 4. MVP機能要件

### 参加者向け

- **認証**: ログイン / 新規登録（メール+パスワード）
- **メール認証**: 新規登録時にメールアドレスの本人確認を実施
- **パスワードリセット**: メールによるパスワード再設定
- **規約同意**:
  - 会員: 初回ログイン時に表示（写真撮影同意を含む / アプリ内固定テキスト）。`users.termsAgreedAt` と `termsAgreedVersion` に記録
  - ゲスト: 参加登録フォームで必須チェックボックスとして表示。`registrations.termsAgreedAt` と `termsAgreedVersion` に記録
  - 規約テキストは**日英2バージョン**を用意し、ユーザーの言語設定に応じて表示（法的明確化のため）
  - **規約バージョン更新時**: `termsAgreedVersion < 現行バージョン` のユーザーに対し再同意画面を表示。合意後に新バージョンを記録
- **マイページ**（要ログイン）
  - プロフィール表示・編集（任意項目は後から入力可能 / メール変更時は Stripe Customer の email も同期更新）
  - 参加予定イベント
  - 参加回数・履歴
- **新規登録の入力項目**:
  - **必須**: email / password / name
  - **任意**（マイページで後から入力可能）: birthDate / attribute / nationality / residenceCountry / residenceCity / preferredLanguage
- **イベント一覧・詳細閲覧**（ゲストも可）
  - 残り枠数の表示
  - 開催場所の地図リンク（Google Maps等）
- **イベント参加登録**:
  - 会員: ログイン状態から登録
  - ゲスト: メール + 規約同意で登録。名前は自動生成（Guest123等）。当日現金払いのみ対応
- **ゲスト登録**:
  - イベント詳細ページから「ゲストとして参加」ボタン
  - または会場のQRスキャン後に「ゲスト登録」リンク（未ログイン時）
  - メールアドレス・規約同意のみで登録可能。即座に参加確定
  - stripe_onlyイベントはゲスト非対応（ボタン非表示）
- **イベント参加キャンセル**: 開催開始時刻までキャンセル可能
  - **返金期限内**（`events.refundDeadline`以前）にキャンセル: `events.refundable = true` かつ Stripe決済済みの場合は自動返金（全額）
  - **返金期限後**にキャンセル: キャンセルは可能だが返金なし
  - ゲストは登録完了メール内のキャンセルリンクから操作可能
- **当日飛び込み参加**: その場でアプリから参加登録が可能（`endDateTime`まで受付。定員到達時は運営の手動オーバーライドで受付可能）
- **決済**: Stripe Checkout による事前決済 / 当日現金払い（イベントごとに事前決済限定の設定が可能）
- **セルフチェックイン**: 会場に掲示されたQRコードをスマホでスキャン（会員のStripe事前決済のみ対応）。スキャン後、運営が最終確認を行う（ゲート無しのため）
  - 対象: 会員 + Stripe事前決済 + paymentStatus='paid'
  - 非対象: ゲスト / 会員でも当日現金払い / unpaid状態
- **支払いステータス表示**: 未払い / 支払い済み
- **メール通知**: 参加登録確認・リマインド（イベント前日に自動送信）・ゲストへのキャンセル用リンク
  - **システム部分の文言**（件名・定型文等）はユーザーの言語設定に応じて日英切替
  - **イベント情報部分**（title / description / location）はDB上の混在テキストをそのまま埋め込み
- **ゲスト→会員引き継ぎ**: ゲストで参加登録した後に会員登録した場合、メールアドレスで過去の参加履歴を自動紐付け
- **多言語対応**: 日本語 / 英語
  - **UI**: ボタン・メニュー・システムメッセージ等は i18n で日英切替（next-i18next 等を想定）
  - **イベント情報**: `title` / `description` / `location` は単一フィールドで**日英混在運用**を想定（現コミュニティ運用の継続）
  - **日時表示**: 言語設定に連動（`2026/04/11` / `Apr 11, 2026`）
  - **言語選択**: `users.preferredLanguage` に保存。未ログイン時はブラウザ設定を自動検出

### 運営向け

- **イベント管理**: 一覧 / 詳細 / 作成 / 編集 / 複製 / 受付停止
- **運営スタッフ割当**: 各イベントに担当運営スタッフ（管理者ユーザー）を割り当て（1イベントあたり3名程度を想定）。割り当てられたスタッフは管理画面・受付専用画面で確認可能
- **定員管理**: イベントごとに参加上限を設定
  - **定員到達時**: 「満員のため受付を締め切りました」と表示（キャンセル枠が発生したら自動で再受付）
  - **運営による手動受付停止**: 「受付を締め切りました」と表示（定員到達とは別表記）
  - **定員超過許可**: 運営が手動でオーバーライドすることで、定員を超えた受付を許可可能
- **返金ポリシー設定**: イベントごとに返金可否を設定（キャンセル時の自動返金を制御）
- **参加者管理**: イベントごとの参加者一覧・支払いステータス管理（未払い / 支払い済み）・参加キャンセル処理
- **登録者一覧**: 全ユーザーの一覧
- **受付専用画面**: チェックイン用QRコードの表示 + 名前検索による手動チェックイン（ゲスト含む） + リアルタイム受付状況をワンストップで操作できるイベント当日向け画面。複数運営者が同時操作可能
- **リアルタイム受付状況**: 登録数 / チェックイン済み / 未チェックインをリアルタイム表示。チェックイン時にアプリ内で運営へ通知
- **未払いリマインド**: 未払い参加者への一括リマインドメール送信
- **売上確認**: イベントごとの売上を表示（`SUM(feeAtRegistration WHERE paymentStatus='paid') - SUM(refundedAmount WHERE refundedAt IS NOT NULL)` で算出）
- **現金返金の手動記録**: 管理画面から現金払い分の返金を手動で記録可能（返金日時・返金金額を入力 → `registrations.refundedAt` / `refundedAmount`に保存）
- **イベント自動クローズ**: `endDateTime` を過ぎたイベントのステータスを自動でclosedに変更
- **メール通知受信**: 参加申込時の通知を**運営共有アドレス**に送信（送信先アドレスは環境変数で設定）
- **運営ロール管理**: MVPでは運営ロール（admin）の付与はSupabaseダッシュボードから手動実施。アプリ内管理UIはフェーズ2以降で実装予定
- **運営操作の監査ログ**: 重要な運営操作を `admin_audit_logs` テーブルに自動記録
  - 記録対象: 手動チェックイン / 支払いステータス変更 / 現金返金記録 / 定員オーバーライド / 受付停止・再開 / 参加キャンセル（運営操作） / イベントキャンセル / 運営ロール変更
  - 記録項目: 操作者 / アクション種別 / 対象エンティティ / 変更内容 / IPアドレス / 日時
  - MVPでは**閲覧UIなし**、Supabaseダッシュボードから直接クエリで確認
  - 保持期間は無期限（将来的にデータ量に応じて自動クリーンアップを検討）

## 5. 技術構成

| レイヤー | 技術 |
| --- | --- |
| フロントエンド | Next.js（PWA） |
| 認証 | Supabase Auth（メール/パスワード） |
| データベース | Supabase（PostgreSQL） |
| サーバーサイド処理 | Supabase Edge Functions / Next.js API Routes |
| 決済 | Stripe Checkout |
| ファイルストレージ | Supabase Storage |
| メール通知 | Supabase Edge Functions 経由 |
| ホスティング | Vercel |

**技術方針:**

- MVP → 検証 → 改善のサイクル
- AI駆動開発 + 重要ロジックにはテスト実装
- 可用性重視（落ちないことが最優先）

**テスト対象（重要ロジック）:**

- 支払いステータス管理
- チェックイン処理（QRセルフ + スタッフ手動）
- QRコード生成・認証
- 返金処理（Stripe連携）
- ゲスト→会員の履歴引き継ぎ
- 受付可否判定ロジック（定員・キャンセル枠・オーバーライド）
- キャンセル・返金期限の判定ロジック
- 監査ログの記録（重要操作が漏れなく記録されるか）

### QRコード仕様（セルフチェックイン）

**方式:** 静的QRコード + サーバー側での時間窓検証

**QRコードの中身:**

```
https://app.example.com/checkin/{eventId}?token={署名済みトークン}
```

- 署名済みトークンは `HMAC-SHA256(eventId + secret)` で生成
- イベント作成時に1回生成し、`events.checkinToken`に保存
- 会場にはA4などで印刷して掲示（運営の端末不要）

**サーバー側の検証ルール（すべて満たす必要あり）:**

0. ログイン済みであること（userId存在）
   - 未ログイン: ログイン画面へリダイレクト（元QR URLを保持）
   - ゲスト希望時: ログイン画面に「ゲストとして登録」リンクを表示
1. `token`がHMAC検証で正しい
2. 現在時刻が **開催開始30分前 〜 開催終了時刻** の範囲内
3. ユーザーが該当イベントに登録済み（`registrations`に有効なレコードあり）
4. `paymentStatus = 'paid'`
5. `paymentMethod = 'stripe'`（現金払いはセルフチェックイン不可）
6. `checkedIn = false`（二重チェックイン防止）
7. スタッフによる最終確認待機状態へ遷移（自動的に`checkedIn = true`にはならない）

**不正時の挙動:**

- 未ログイン: ログイン/ゲスト登録画面へリダイレクト
- 時間窓外: 「チェックイン可能時間外です」
- 未決済: 「未決済です。運営にお声がけください」
- 現金払い: 「現金払いの場合は運営にお声がけください」
- 未登録: 「このイベントに登録されていません」
- 二重チェックイン: 「既にチェックイン済みです」

**スタッフ最終チェックイン（全ユーザー）:**

ゲートがないため、スタッフによる目視確認 + 本人確認を必須とする。

- **QRセルフチェックイン後**: ユーザーがQRを有効にしたら、受付画面に「チェックイン準備完了」状態で表示。スタッフが本人確認 → 「チェックイン完了」ボタン
- **ゲスト・現金払いユーザー**: スタッフ受付画面でメアド/登録情報を検索 → 本人確認 → 現金受取（必要な場合）→ 「チェックイン完了」ボタン
- 最終チェックイン時に `registrations.checkedIn = true`, `checkedInAt = 現在時刻` を記録
- 監査ログに `manual_checkin` として記録（QR準備完了経由でも、スタッフが最終確認したら記録）

## 6. データ設計

### ユーザー (users)

| フィールド | 内容 | 必須/任意 |
| --- | --- | --- |
| id | Supabase Auth UUID | 必須 |
| email | メールアドレス | 必須 |
| name | 名前 | 必須 |
| birthDate | 生年月日（年齢は算出で取得） | 任意 |
| attribute | 属性 enum: `student` / `worker` / `entrepreneur` / `other` | 任意 |
| nationality | 国籍（ISO 3166-1 alpha-2 コードの配列 / 複数選択可） | 任意 |
| residenceCountry | 居住国（ISO 3166-1 alpha-2 コード / 単一値） | 任意 |
| residenceCity | 居住都市・地域（自由記述テキスト / 例: `福岡市`, `Seoul`） | 任意 |
| preferredLanguage | 言語設定（`ja` / `en` / デフォルト `ja`） | 任意 |
| stripeCustomerId | Stripe Customer ID（`cus_xxxxx` / null = 未作成） | 任意（初回決済時に自動作成） |
| role | ロール（`participant` / `admin`） | 必須（デフォルト `participant`） |
| termsAgreedAt | 規約同意日時 | 必須（初回ログイン時に記録） |
| termsAgreedVersion | 同意した規約バージョン（例: `1.0`, `2.0`） | 必須（初回ログイン時に記録） |
| createdAt | 登録日時 | 必須 |

**各enumの値:**

- **attribute**: `student`（学生）/ `worker`（社会人）/ `entrepreneur`（起業家）/ `other`（その他）
  - 将来的に選択肢の追加が想定される
- **nationality**: ISO 3166-1 alpha-2 の国コード（例: `JP`, `US`, `CN`, `KR` 等）を**配列**で保持。複数国籍対応
- **residenceCountry**: ISO 3166-1 alpha-2 の国コード（単一値）

**住居情報の入力UI:**

- `residenceCountry` は ISO国リストからプルダウン選択（日本がデフォルト候補）
- `residenceCity` は自由記述テキスト
- `residenceCountry = 'JP'` の場合のみ、以下の**クイック選択チップ**を表示して入力を補助（人口上位5市 + その他）:
  - 福岡市 / 北九州市 / 久留米市 / 飯塚市 / 大牟田市 / その他
  - チップを押すとテキストフィールドに都市名が自動入力される
  - 「その他」を押すとテキストが空になり自由入力モードに

**プロフィール入力の運用:**

- 新規登録時は `email` / `password` / `name` のみ必須
- 任意項目は登録後にマイページで編集可能
- 段階的な入力を許容し、新規登録の離脱を減らす

**Stripe Customer の運用:**

- **会員のみ** Stripe Customer を作成し `users.stripeCustomerId` に保持（ゲストは作成しない）
- 会員の**初回決済時**に Stripe Customer を自動作成
  - `email`, `name` を渡して Customer を作成
  - `users.stripeCustomerId` に保存
  - Stripe Checkout Session に `customer` パラメータを指定
- **2回目以降の決済**は保存済みの `stripeCustomerId` を使用（決済履歴がStripe側で集約される）
- **ゲスト決済時**は Customer を作成せず、Stripe Checkout Session に `customer_email` のみ渡す
- **ユーザーがメールアドレスを変更した場合**、`stripeCustomerId` が存在すれば Stripe API で Customer の email も同期更新する
- **ゲスト→会員引き継ぎ時**は Stripe Customer を新規作成せず、次回の決済時に作成される（過去のゲスト決済は Customer 紐付けなしのまま残る）

### イベント (events)

| フィールド | 内容 |
| --- | --- |
| id | イベントID |
| title | タイトル |
| startDateTime | 開催開始日時 |
| endDateTime | 開催終了日時 |
| location | 開催場所 |
| description | 説明文 |
| fee | 参加費 |
| capacity | 定員 |
| imageUrl | カバー画像URL |
| status | ステータス（draft / published / closed / cancelled） |
| registrationOpen | 受付中フラグ（運営が手動で停止可能） |
| paymentType | 決済方式（stripe_only / stripe_and_cash） |
| refundable | 返金可否フラグ（キャンセル時の自動返金を制御） |
| refundDeadline | 返金期限（デフォルト: `startDateTime - 24時間`）。この日時以降のキャンセルは返金対象外 |
| allowOverCapacity | 定員超過許可フラグ（運営の手動オーバーライド / デフォルトfalse） |
| checkinToken | チェックインQR検証用の署名済みトークン（HMAC-SHA256） |
| createdBy | 作成者のユーザーID（users.role = 'admin'） |
| createdAt | 作成日時 |

**チェックイン可能時間窓:** `startDateTime - 30分` 〜 `endDateTime`

**受付可否判定ロジック:**

```
有効な登録数 = registrations.count(WHERE eventId = X AND cancelledAt IS NULL)

受付可能 = (
  events.registrationOpen = true AND
  events.status = 'published' AND
  (有効な登録数 < events.capacity OR events.allowOverCapacity = true) AND
  現在時刻 < events.endDateTime
)
```

- `registrationOpen = false`（運営が手動停止）: 「受付を締め切りました」と表示
- `有効な登録数 >= capacity` かつ `allowOverCapacity = false`: 「満員のため受付を締め切りました」と表示
- キャンセルで有効な登録数が減少した場合は自動的に再受付となる

**キャンセル・返金の判定ロジック:**

```
キャンセル可否:
  現在時刻 < startDateTime かつ cancelledAt IS NULL
  → キャンセル可能

キャンセル成立時の返金可否:
  events.refundable = true AND
  現在時刻 < events.refundDeadline AND
  paymentStatus = 'paid'
  → 返金対象

返金実行:
  paymentMethod = 'stripe': Stripe APIで自動返金、refundedAt に日時、refundedAmount に `feeAtRegistration` を記録
  paymentMethod = 'cash': 運営が管理画面から手動で返金記録を入力（refundedAmount = `feeAtRegistration`）
```

### 参加履歴 (registrations)

| フィールド | 内容 |
| --- | --- |
| id | 登録ID |
| userId | ユーザーID（ゲストの場合は null） |
| guestName | 自動生成ゲスト名（例: Guest123 / 会員の場合は null） |
| guestEmail | ゲストメールアドレス（会員の場合は null） |
| guestCancelToken | ゲスト用キャンセルリンクの署名済みトークン（会員の場合は null） |
| eventId | イベントID |
| paymentStatus | 支払いステータス（unpaid / paid） |
| paymentMethod | 支払い方法（stripe / cash） |
| stripePaymentId | Stripe決済ID（返金処理用 / nullの場合は現金払い） |
| feeAtRegistration | 登録時点の参加費（円 / events.fee をコピー）。料金変更後の返金額計算に使用 |
| refundedAt | 返金日時（null = 未返金） |
| refundedAmount | 返金金額（円 / null = 未返金）。常に`feeAtRegistration`に基づく全額返金。将来の部分返金対応と監査のため記録 |
| checkedIn | チェックイン済みフラグ |
| checkedInAt | チェックイン日時 |
| cancelledAt | キャンセル日時（null = 有効） |
| termsAgreedAt | 規約同意日時（ゲスト登録時に記録 / 会員は users.termsAgreedAt を参照） |
| termsAgreedVersion | 同意した規約バージョン（例: `1.0` / ゲスト登録時に記録） |
| createdAt | 登録日時 |

**制約:** `userId IS NOT NULL` または `(guestName IS NOT NULL AND guestEmail IS NOT NULL)` のいずれかを満たすこと

**ゲスト→会員引き継ぎロジック:**

1. 会員新規登録時、同じメールアドレスの`registrations`レコード（`guestEmail`が一致）を検索
2. 該当レコードの`userId`を新ユーザーのUUIDに更新
3. `guestName`・`guestEmail`・`guestCancelToken`をNULLに更新

### 運営操作の監査ログ (admin_audit_logs)

重要な運営操作の証跡を記録するテーブル。

| フィールド | 内容 |
| --- | --- |
| id | ログID |
| adminUserId | 操作を行った運営のユーザーID（`users.id`） |
| action | アクション種別（下記enum） |
| targetType | 対象エンティティ種別（`registration` / `event` / `user`  等） |
| targetId | 対象エンティティのID |
| details | 変更内容のJSON（例: `{"before": "unpaid", "after": "paid"}`） |
| ipAddress | 操作時のIPアドレス（HTTPリクエストヘッダから取得） |
| createdAt | 操作日時 |

**actionのenum値:**

- `manual_checkin`: 手動チェックイン
- `payment_status_update`: 支払いステータス手動変更（現金受領等）
- `cash_refund_record`: 現金返金の手動記録
- `capacity_override`: 定員超過オーバーライドの切替
- `registration_toggle`: 受付停止/再開
- `registration_cancel_by_admin`: 運営による参加キャンセル
- `event_cancel`: イベントキャンセル
- `admin_role_change`: 運営ロール付与/剥奪
- （将来的に追加予定）

**記録例:**

```json
{
  "adminUserId": "admin-uuid-123",
  "action": "payment_status_update",
  "targetType": "registration",
  "targetId": "reg-uuid-456",
  "details": {
    "before": { "paymentStatus": "unpaid" },
    "after": { "paymentStatus": "paid", "paymentMethod": "cash" },
    "note": "当日現金受領"
  },
  "ipAddress": "203.0.113.1",
  "createdAt": "2026-04-25T19:35:12Z"
}
```

**運用方針:**

- MVPでは専用の閲覧UIを提供せず、Supabaseダッシュボードから直接クエリで確認
- フェーズ2以降で管理画面に「監査ログ閲覧」画面を追加予定（フィルタ・検索機能付き）
- 保持期間は**無期限**（小規模コミュニティではデータ量は月数十件程度を想定。データ量が増えた場合は自動クリーンアップを検討）

### 運営スタッフ割当 (event_staff)

各イベントに担当運営スタッフを割り当てる中間テーブル。

| フィールド | 内容 |
| --- | --- |
| id | 割当ID |
| eventId | イベントID |
| userId | スタッフのユーザーID（users.role = 'admin' のみ） |
| assignedAt | 割当日時 |

**制約:** `(eventId, userId)` の組み合わせはユニーク

**運用想定:**

- 1イベントあたり3名程度の運営スタッフを割り当て
- 割り当てられたスタッフは管理画面・受付専用画面で確認可能
- 当日の運営メンバーを事前に明示化できる

### 個人情報の方針

- 取得は最小限にとどめる
- クレジットカード情報はアプリ側で保持しない（Stripeに委託）
