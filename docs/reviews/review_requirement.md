## 🔴 矛盾・要決定（最優先）

### 1. ゲストPath A の当日QRスキャン時のOTP認証フロー欠落 ✅ 対応済

- **該当箇所:**
  - `flows.md` L52: ゲストが当日QRスキャン → 「OTP認証（未ログインの場合）」
  - `business-rules.md` L51 「未ログイン時の挙動」: 「ログイン画面へリダイレクト → ログイン / ゲストとして参加 / 新規会員登録」の3リンク
- **矛盾:** Path A で事前登録済みのゲストが当日スキャンしたとき、ログイン画面に飛んでも「ログインできない」「再度ゲスト登録するのは重複」「新規会員登録は別人扱い」 → どこにも入れない
- **対応:** ゲスト事前登録を MVP スコープ外に確定（`out-of-scope.md` L20）。当日QR経由の新規ゲスト登録のみとなり、本矛盾は構造的に消滅。`business-rules.md` L51 の「ゲストとして参加」リンクは `?from=qr` 時のみ表示する設計で整合。

### 2. イベントキャンセル通知のタイミング矛盾 ✅ 対応済

- **該当箇所:**
  - `business-rules.md` L123: 「参加者への通知は返金確定後に送信」
  - `business-rules.md` L126: 「全参加者へイベントキャンセル通知メールを自動送信」
- **矛盾:** 返金失敗者は手動対応待ちのため、通知が当日まで遅延するリスク
- **対応:** 案C採用 + 運営側集計レポート追加
  - 参加者: キャンセル確定即時に通知送信。返金状況は文言で出し分け（成功/失敗/非対象）
  - 運営: 返金集計結果（成功数/失敗数）+ 失敗者リスト付きメール送信
  - `business-rules.md` L120-134 に詳細記載

### 3. `paymentStatus = paid` + `refundStatus = failed` の状態表記不整合 ✅ 対応済

- **問題:** イベントキャンセル時に返金失敗 → paymentStatus は paid のまま、refundStatus = failed という中間状態
- **影響:**
  - 売上集計（`paymentStatus = 'paid'`）に含まれる
  - ユーザーマイページで「支払い済み」と表示される（実態は中止イベントの未返金）
- **対応:**
  - refundStatus で細分化: `null / pending / succeeded / failed`
  - マイページUI: 返金処理中 / 返金失敗を明示表示（運営連絡先付き）
  - 売上集計: `paymentStatus='paid' AND (refundStatus IS NULL OR refundStatus='failed')`
  - 詳細: `business-rules.md` L67-79（マイページ表示ルール）/ `features.md` L92-93（売上計算）

### 4. ゲスト→会員引き継ぎ時の規約同意状態 ✅ 対応済

- **問題:**
  - ゲスト時の同意は `registrations.termsAgreedAt` に記録
  - 引き継ぎで `userId` 紐付けしても `users.termsAgreedAt` は NULL のまま
  - → 「強制リダイレクトルール」で再同意要求される（既同意済みなのに）
- **対応:** ゲスト→会員アカウント引き継ぎを MVP スコープ外に確定（`out-of-scope.md` L18）。引き継ぎ機構自体が無いため矛盾は消滅。フェーズ2で再開時に同意情報コピーの仕様を追加すること。

## 🟠 機能・要件の欠落

### 5. Stripe Webhook 処理 ✅ 対応済

- **欠落:** 決済確認の仕組みが完全に未記述
- **対応:** Webhook-driven フロー + サーバーダウン対策 Cron 同期を追加
  - Webhook: `checkout.session.completed` → 署名検証 → 冪等性処理 → `paymentStatus='paid'` 更新
  - Cron同期: 毎時実行 → `paymentStatus='pending'` で 1時間以上放置 → Stripe API 再確認 → 更新
  - フィールド追加: `stripeSessionId`（Cron用）/ `stripePaymentId`（返金用）分離
  - 詳細: `business-rules.md` L138-157（Stripe決済確認フロー）

### 6. 受付専用画面のアクセス制御 ✅ 対応済

- **不明確:** `event_staff` に割り当てられたスタッフのみ？ 全 admin？
- **対応:** 案B採用（全admin アクセス可）
  - アクセス権: `users.role = 'admin'` なら全員可
  - event_staff 割当: アクセス制限ではなく運営チーム識別用（操作履歴に名前付与）
  - 詳細: `features.md` L77-84

### 7. 無料イベント（`fee = 0`）の扱い ✅ 対応済

- **欠落:** events.fee = 0 の場合の決済フロー未定義
- **対応:**
  - Stripe Checkout スキップ → 参加登録時に直接 `paymentStatus='paid'` 設定
  - `paymentMethod='free'` 追加（stripe / cash / free の3値に）
  - 返金ロジックなし（fee が 0 なため）
  - QR セルフチェックイン可能（事前決済済み扱い）
  - 詳細: `business-rules.md` L138-153（無料イベント）

### 8. ゲスト OTP ロックアウト時の運営介入手段 ✅ 対応済

- **問題:** ゲストがメアド誤入力で登録 → OTP届かず → マイページアクセス不可、キャンセル不可
- **対応:** 案A採用（削除のみ）
  - 管理画面から当該ゲスト登録を削除
  - ゲストが会場で再度 QR スキャン → 新規登録フロー開始（メアド再入力可）
  - シンプル設計（メアド修正等の複雑な修復不要）
  - 詳細: `business-rules.md` L155-164（ゲスト参加管理）

### 9. 既チェックイン者のキャンセル ✅ 対応済

- **欠落:** イベント運営都合キャンセル時、既に `checkedIn = true` の参加者の扱い
- **対応:**
  - チェックイン済みは返金対象外（既に参加済み）
  - `cancelledAt` 更新なし（参加事実は残す）
  - 通知メール対象外（会場にいる想定）
  - 返金は未済参加者のみ実施
  - 詳細: `business-rules.md` L192-219（イベント運営都合キャンセル）

### 10. ゲスト名の重複問題 ✅ 対応済

- **問題:** 「Guest123」自動生成名が同一イベント内で重複する可能性
- **対応:**
  - ゲスト名を `Guest-{イベント内シーケンス}` に変更（Guest-1, Guest-2, ...）
  - イベント単位でカウンタ管理 → 重複なし
  - 受付画面: メアド一部表示（`h***@example.com`）で補助識別
  - 検索: 名前またはメアドで検索可能（メアドで一意特定）
  - 詳細: `business-rules.md` L155-163（ゲスト名の自動生成）

### 11. `events.fee` 変更ポリシー ✅ 対応済

- **欠落:** 既存登録者がいるイベントで fee 変更可能か
- `feeAtRegistration` で履歴は保護されるが、運営UI側の制約・警告が未定義
- **対応:** 案B採用（警告のみ）
  - Fee 変更は常に可能（既登録者は保護済み）
  - 既登録者がいる場合、変更前に警告表示：「〇名の既登録者がいます。既登録者の請求額は変わりません」
  - 新規登録者のみ新 fee 適用
  - 詳細: `business-rules.md` L252-259（参加費の変更ポリシー）

## 🟡 運用面の欠落

### 12. 自動処理の実装方法 ✅ 対応済

- **欠落:** 以下の自動処理の実装基盤が未記述
  - イベント自動クローズ（`endDateTime`経過）
  - リマインドメール送信（`reminderHoursBefore`）
- **対応:** ハイブリッド構成採用 + デプロイ先 Cloudflare 移行
  - **Supabase Cron（pg_cron）**: イベント自動クローズ（純SQL）
  - **Cloudflare Workers Cron Triggers**: リマインドメール送信、Stripe 決済ステータス同期（外部API + テンプレート + 多言語切替）
  - 二重送信防止: `registrations.reminderSentAt` フィールド追加
  - リマインド対象: 会員のみ（ゲストは当日登録のため対象外）
  - 詳細: `tech.md`（技術構成）/ `business-rules.md` L284-301（自動処理の振り分け）

### 13. PWA 要件 ✅ 対応済

- **欠落:** `tech.md` に「Next.js（PWA）」とラベルのみ。具体要件未記述
- **対応:** 軽量 PWA 採用（フルオフライン非対応）
  - **採用ライブラリ**: `vite-plugin-pwa`（Workbox）
  - **キャッシュ戦略**:
    - 静的アセット: Cache-first（precaching）
    - API: Network-only（リアルタイム性優先）
    - 画像: Stale-while-revalidate
  - **インストール可能性**: manifest.json + 参加者マイページでインストールプロンプト表示
  - **非対応**: オフライン操作 / バックグラウンド同期 / プッシュ通知（フェーズ2）
  - **受付画面**: オンライン前提（Realtime 依存）
  - 詳細: `tech.md` PWA 方針セクション
